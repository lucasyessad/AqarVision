M00 — Foundations


Objectif


Poser les fondations techniques, structurelles et organisationnelles du produit pour éviter toute dette d’architecture dès le départ.


Périmètre


Ce module ne crée pas de valeur fonctionnelle directe pour l’utilisateur final. Il structure le produit pour permettre le développement rapide, cohérent et maintenable de tous les modules suivants.


Il couvre :

  • monorepo

  • conventions de code

  • stratégie i18n

  • design system

  • feature flags

  • observabilité de base

  • stratégie environnements


Structure cible

apps/
  web/
  mobile/
packages/
  domain/
  database/
  ui-web/
  ui-mobile/
  security/
  analytics/
  feature-flags/
  config/

Décisions structurantes


Le projet est un modular monolith.

Cela signifie :

  • une seule application métier cohérente

  • séparation claire des domaines

  • base centralisée

  • pas de microservices au départ


Pourquoi :

  • coût opérationnel plus faible

  • vitesse d’exécution plus élevée

  • moins de couplage réseau

  • meilleure cohérence métier


i18n


Les langues sont actives dès le départ :

  • fr

  • ar

  • en

  • es


Contraintes :

  • toutes les routes sont préfixées par locale

  • l’arabe active automatiquement le RTL

  • les composants ne doivent jamais coder des marges gauche/droite en dur

  • tous les textes statiques passent par un système de traduction


Feature flags


Un registre central permet d’activer ou désactiver :

  • billing

  • IA premium

  • analytics premium

  • maps premium

  • push premium


Observabilité de base


Dès M00 :

  • logs structurés

  • corrélation des requêtes

  • suivi erreurs runtime

  • health checks

  • environment validation au démarrage


Livrables

  • structure repo figée

  • conventions de dossiers

  • stratégie noms de fichiers

  • conventions TypeScript

  • conventions Zod

  • tokens design

  • config i18n

  • registre feature flags


Critères d’acceptation

  • la structure du repo est figée

  • aucune feature future ne doit nécessiter de refonte du socle

  • les 4 langues sont supportées

  • le support RTL est validé

  • le système de flags fonctionne

  • les environnements sont clairement séparés


Tests

  • validation chargement locale

  • test RTL visuel

  • test fallback de traduction

  • test lecture flags


Dépendances


Aucune.


Risques

  • sous-estimer l’impact du RTL

  • multiplier les conventions contradictoires

  • ne pas figer le monorepo assez tôt

M02 — Agencies, Branches & Team


Objectif


Modéliser l’organisation professionnelle.


Périmètre

  • branches multi-villes

  • équipe

  • invitations

  • rôles agence


Données

  • agency_branches

  • agency_memberships


Modèle métier


Une agence peut :

  • avoir plusieurs branches

  • être présente dans plusieurs wilayas

  • avoir plusieurs membres

  • attribuer des rôles différents selon les utilisateurs


Rôles

  • owner

  • admin

  • agent

  • editor

  • viewer


Règles métier

  • une agence a au moins un owner

  • une branche appartient à une seule agence

  • un utilisateur peut appartenir à plusieurs agences

  • les permissions dépendent du couple (user_id, agency_id)


API / actions

  • createAgency

  • updateAgency

  • createBranch

  • updateBranch

  • inviteMember

  • changeMembershipRole

  • deactivateMembership


UI

  • création agence

  • gestion branches

  • liste des membres

  • gestion rôles

  • écran d’invitation


Sécurité

  • seule une personne autorisée peut modifier l’agence

  • un viewer ne peut pas éditer

  • un admin ne peut pas usurper une autre agence

  • toute lecture/écriture doit être limitée au périmètre agency_id


Critères d’acceptation

  • une agence peut avoir plusieurs branches

  • les rôles sont effectifs

  • l’isolation inter-agence est totale

  • un owner peut administrer les membres


Tests

  • création agence

  • ajout branche

  • rôle owner/admin/agent/viewer

  • accès illégal interdit


Dépendances

Risques

  • confusion entre branch-level et agency-level

  • erreurs RLS sur memberships
