## 1. IDENTITÉ DU PROJET

**AqarVision** est un écosystème proptech algérien composé de :

- **AqarSearch** — Marketplace publique immobilière (recherche, carte, SEO, alertes, favoris)
- **AqarPro** — CRM pour agences/promoteurs (publication, leads, messagerie, analytics, IA)
- **Data Layer** — Infrastructure data (historique prix, domain events, analytics, estimation future)

### Modèle métier

- **Multi-tenant** : chaque agence est un tenant isolé via RLS Postgres
- **Hybride SaaS + marketplace** : côté pro (gating par plan) + côté public (ouvert)
- **Multilingue** : FR / AR / EN / ES — l’arabe active le RTL
- **Marché cible** : Algérie (58 wilayas, 1541 communes)
- **Monétisation** : abonnements agences (Starter/Pro/Enterprise) via Stripe

### Surfaces applicatives

```
Marketing site  →  Auth  →  Dashboard Agence (AqarPro)
                              ↓
                   Publication annonces
                              ↓
                   Marketplace (AqarSearch) → /search + carte
                              ↓
                   Page détail bien  →  Contact (chat + WhatsApp)
                              ↓
Dashboard Agence → Billing (abonnement + gating)
                 → IA (rédaction / traduction)
                 → Analytics
```

|CI/CD          |GitHub Actions                                             |-                       |

### Décisions architecturales

- **Modular monolith** — pas de microservices au départ
- **Server Actions** pour toutes les mutations (CRUD, upload URL, IA, messaging)
- **Route Handlers** uniquement pour callbacks externes (webhooks Stripe)
- **Supabase Auth SSR** via `@supabase/ssr` (cookies, pas localStorage)
- **RLS partout** — deny by default, autoriser explicitement chaque cas
- **PostGIS** pour toutes les requêtes géospatiales (rayon, bounding box, proximité)
- **Signed Upload URLs** pour les médias (upload direct navigateur, bucket privé)
- **ISR/SSR** pour les pages SEO marketplace
- **Feature flags** en DB pour gater les fonctionnalités premium

## 4. CONVENTIONS DE CODE

### Nommage fichiers

- **Fichiers utilitaires** : `kebab-case.ts` (ex: `create-listing.action.ts`)
- **Composants React** : `PascalCase.tsx` (ex: `ListingCard.tsx`)
- **Suffixes obligatoires** :
  - `*.service.ts` — logique métier
  - `*.action.ts` — server actions
  - `*.schema.ts` — validation Zod
  - `*.types.ts` — types/interfaces
  - `*.mapper.ts` — transformation data
  - `*.policy.ts` — règles d’autorisation métier

### Structure feature

Chaque feature suit :

```
feature-name/
  actions/          # Server Actions (orchestration)
  components/       # UI React
  hooks/            # Custom hooks
  schemas/          # Zod validation
  services/         # Logique métier (coordination)
  types/            # Contrats TypeScript
  utils/            # Helpers purs
```

### Règles TypeScript

- `strict: true` partout
- Pas de `any` — utiliser `unknown` + type guards
- Tous les inputs validés via Zod avant traitement
- Tous les outputs typés avec des DTO explicites
- Jamais exposer des colonnes internes inutiles au client

### Pattern Server Action

```typescript
// 1. Valider input via Zod
// 2. Résoudre l'acteur courant (auth)
// 3. Vérifier les droits (policy)
// 4. Appeler le service domaine
// 5. Retourner un output sanitisé
// 6. Déclencher revalidation si nécessaire

"use server";

import { z } from "zod";
import { createServerClient } from "@/lib/supabase/server";
import { listingService } from "@packages/domain/listings";

const CreateListingSchema = z.object({
  agency_id: z.string().uuid(),
  listing_type: z.enum(["sale", "rent", "vacation"]),
  property_type: z.enum(["apartment", "villa", "terrain", "commercial", "office", "building", "farm", "warehouse"]),
  current_price: z.number().nonnegative(),
  wilaya_code: z.string().min(1),
  // ...
});

export async function createListingAction(input: z.infer<typeof CreateListingSchema>) {
  const validated = CreateListingSchema.parse(input);
  const supabase = await createServerClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) throw new Error("UNAUTHORIZED");

  const result = await listingService.create(supabase, user.id, validated);
  return { listing_id: result.id, status: "draft" };
}
```

### Règles interdites

- **JAMAIS** appeler la DB directement depuis un composant React
- **JAMAIS** de logique métier dans les composants UI
- **JAMAIS** de `localStorage` pour la session (cookies only via Supabase SSR)
- **JAMAIS** de logique de rôle uniquement côté frontend
- **JAMAIS** coder des marges left/right en dur (utiliser propriétés logiques pour RTL)
- **JAMAIS** de secrets/tokens dans les logs

### Gestion d’erreurs

```typescript
// Retour standard
type ActionResult<T> = 
  | { success: true; data: T }
  | { success: false; error: { code: string; message: string } };

// Erreurs métier : explicites, lisibles
// Erreurs sécurité : NON bavardes (pas de détails internes)
// Erreurs validation : structurées (champs + messages)
// Erreurs critiques : loggées séparément, pas propagées au client
```

### Logging

```typescript
// Logger : contexte, ids, agency_id, type événement, résultat
// NE PAS logger : secrets, tokens, données sensibles brutes
logger.info("listing.created", {
  listing_id: result.id,
  agency_id: input.agency_id,
  actor: user.id,
});
```

-- Helper : est-ce que l'utilisateur est membre actif d'une agence ?
create or replace function is_agency_member(_user_id uuid, _agency_id uuid)
returns boolean
language sql
security definer
stable
as $$
  select exists (
    select 1
    from agency_memberships am
    where am.user_id = _user_id
      and am.agency_id = _agency_id
      and am.is_active = true
  );
$$;

-- Helper : est-ce que l'utilisateur est admin (owner ou admin) d'une agence ?
create or replace function is_agency_admin(_user_id uuid, _agency_id uuid)
returns boolean
language sql
security definer
stable
as $$
  select exists (
    select 1
    from agency_memberships am
    where am.user_id = _user_id
      and am.agency_id = _agency_id
      and am.role in ('owner', 'admin')
      and am.is_active = true
  );
$$;

-- Helper : est-ce que l'utilisateur est super_admin ?
create or replace function is_super_admin(_user_id uuid)
returns boolean
language sql
security definer
stable
as $$
  select exists (
    select 1
    from profiles p
    where p.user_id = _user_id
      and p.role = 'super_admin'
  );
$$;

-- Trigger : updated_at automatique
create or replace function update_updated_at()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = now();
  return new;
end;
$$;

-- Bootstrap profil après inscription
create or replace function handle_new_user()
returns trigger
language plpgsql
security definer
as $$
begin
  insert into public.users (id) values (new.id);
  insert into public.profiles (user_id, preferred_locale)
    values (new.id, coalesce(new.raw_user_meta_data->>'preferred_locale', 'fr'));
  return new;
end;
$$;

### Supabase SSR Client

```typescript
// lib/supabase/server.ts
import { createServerClient as createClient } from "@supabase/ssr";
import { cookies } from "next/headers";

export async function createServerClient() {
  const cookieStore = await cookies();
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll(); },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value, options }) =>
            cookieStore.set(name, value, options));
        },
      },
    }
  );
}
```

### PostGIS Search

```sql
-- Recherche par bounding box
select l.*, lt.title, lt.slug
from listings l
join listing_translations lt on lt.listing_id = l.id and lt.locale = $1
where l.current_status = 'published'
  and ST_Within(l.location::geometry, ST_MakeEnvelope($2, $3, $4, $5, 4326))
order by l.published_at desc
limit $6 offset $7;

-- Recherche par rayon (km)
select l.*, ST_Distance(l.location, ST_SetSRID(ST_MakePoint($1, $2), 4326)) as distance_m
from listings l
where l.current_status = 'published'
  and ST_DWithin(l.location, ST_SetSRID(ST_MakePoint($1, $2), 4326), $3)
order by distance_m;
```

      // Sync status
      break;

      // Désactiver

## 12. RÈGLES DE SÉCURITÉ

1. **RLS deny by default** — toute table sans policy = accès interdit
1. **Server Actions uniquement** pour les mutations — jamais de mutation côté client
1. **Supabase Auth SSR** — cookies sécurisés, pas de localStorage
1. **Vérification signature Stripe** — webhook rejeté si signature invalide
1. **Pas de logique de rôle uniquement côté frontend** — toujours vérifier côté serveur
1. **Upload via Signed URLs uniquement** — bucket privé, validation type/taille
1. **Données sensibles** (documents juridiques) : bucket privé, accès RLS, pas d’URL publique permanente
1. **PKCE pour le mobile** — stockage sécurisé des tokens
1. **Rate limiting** : implémenter sur les endpoints publics (search, leads, auth)
1. **Protection CSRF** : activée nativement par Server Actions (comparaison origin/host)
1. **Secrets hors repo** — variables d’env Vercel, jamais dans le code
1. **Audit log** sur toutes les actions admin (modération, changement plan, etc.)

### SEO Technique

- `hreflang` link tags pour les 4 locales
- `canonical` sur chaque page (éviter la duplication)
- Sitemap XML dynamique (listings publiés + pages catégories)
- JSON-LD `RealEstateListing` sur les pages détail
