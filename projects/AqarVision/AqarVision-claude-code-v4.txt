## 1. IDENTITE DU PROJET

**AqarVision** est un ecosysteme proptech algerien compose de :

- **AqarSearch** — Marketplace publique immobiliere (recherche, carte, SEO, alertes, favoris)
- **AqarPro** — CRM pour agences/promoteurs (publication, leads, messagerie, analytics, IA)
- **Data Layer** — Infrastructure data (historique prix, domain events, analytics, estimation future)

### Modele metier

- **Multi-tenant** : chaque agence est un tenant isole via RLS Postgres
- **Hybride SaaS + marketplace** : cote pro (gating par plan) + cote public (ouvert)
- **Multilingue** : FR / AR obligatoires au lancement — EN / ES optionnels (ajoutables par l'agence via IA)
- **Marche cible** : Algerie (58 wilayas, 1541 communes)
- **Monetisation** : abonnements agences (Starter/Pro/Enterprise) via Stripe **en EUR** (le DZD n'est pas supporte par Stripe). Les prix immobiliers sont affiches en **DZD**.

### Strategie monnaie

| Contexte | Devise | Justification |
|----------|--------|---------------|
| Prix des biens immobiliers | DZD | Marche local algerien |
| Abonnements agences (Stripe) | EUR | Stripe ne supporte pas le DZD |
| Affichage pricing page | EUR + equivalent indicatif DZD | Transparence pour les agences |

> **Note** : Le taux de conversion EUR/DZD est indicatif uniquement (affichage). Aucune conversion automatique n'est effectuee. Les abonnements sont debites en EUR sur la carte de l'agence.

### Surfaces applicatives

```
Marketing site  ->  Auth  ->  Dashboard Agence (AqarPro)
                              |
                   Publication annonces
                              |
                   Moderation (pending_review -> published)
                              |
                   Marketplace (AqarSearch) -> /search + carte
                              |
                   Page detail bien  ->  Contact (chat temps reel + WhatsApp)
                              |
Dashboard Agence -> Billing (abonnement + gating)
                 -> IA (redaction / traduction)
                 -> Analytics
```

### Decisions architecturales

- **Modular monolith** — pas de microservices au depart
- **Server Actions** pour toutes les mutations (CRUD, upload URL, IA, messaging)
- **Route Handlers** uniquement pour callbacks externes (webhooks Stripe)
- **Supabase Auth SSR** via `@supabase/ssr` (cookies, pas localStorage)
- **Supabase Realtime** pour le messaging en temps reel (presence + broadcast)
- **RLS partout** — deny by default, autoriser explicitement chaque cas
- **PostGIS** pour toutes les requetes geospatiales (rayon, bounding box, proximite)
- **pg_trgm + tsvector** pour la recherche full-text sur les annonces
- **Signed Upload URLs** pour les medias (upload direct navigateur, bucket prive)
- **ISR/SSR** pour les pages SEO marketplace avec invalidation explicite via `revalidateTag`
- **Feature flags** en DB pour gater les fonctionnalites premium
- **Soft-delete** sur les entites critiques (listings, agencies) via `deleted_at`
- **Verrouillage optimiste** sur listings via colonne `version`

### Clarification packages/domain vs src/features/*/services

| Couche | Localisation | Responsabilite | Dependances |
|--------|-------------|----------------|-------------|
| **Domain** | `packages/domain/` | Logique metier **pure** : validation, calculs, regles, policies. Aucune dependance IO (pas de Supabase, pas de fetch). | Uniquement `packages/database/types` |
| **Feature services** | `src/features/*/services/` | **Orchestration** : appelle Supabase, coordonne le domain, gere les side-effects (cache, events). | `packages/domain` + `@supabase/ssr` |
| **Actions** | `src/features/*/actions/` | **Point d'entree** : validation Zod, auth, appelle le service, retourne `ActionResult<T>`. | Feature services |

> **Regle** : Si une fonction peut etre testee sans mock Supabase, elle va dans `packages/domain/`. Sinon, elle va dans `src/features/*/services/`.

### Source de verite migrations

- `supabase/migrations/` est la **seule source de verite** pour le schema DB
- `packages/database/types/` contient les types **generes automatiquement** via `supabase gen types typescript`
- Ne jamais ecrire de types DB manuellement dans `packages/database/`

### Pattern Server Action (CORRIGE)

```typescript
"use server";

import { z } from "zod";
import { revalidateTag } from "next/cache";
import { createServerClient } from "@/lib/supabase/server";
import { listingService } from "@/features/listings/services/listing.service";
import { rateLimit } from "@/lib/rate-limit";
import { logger } from "@/lib/logger";
import type { ActionResult } from "@/lib/utils/action-result";

const CreateListingSchema = z.object({
  agency_id: z.string().uuid(),
  listing_type: z.enum(["sale", "rent", "vacation"]),
  property_type: z.enum(["apartment", "villa", "terrain", "commercial", "office", "building", "farm", "warehouse"]),
  current_price: z.number().nonnegative(),
  wilaya_code: z.string().min(1),
  // ...
});

export async function createListingAction(
  input: z.infer<typeof CreateListingSchema>
): Promise<ActionResult<{ listing_id: string; status: string }>> {
  try {
    // 1. Valider input
    const parsed = CreateListingSchema.safeParse(input);
    if (!parsed.success) {
      return { success: false, error: { code: "VALIDATION_ERROR", message: parsed.error.issues[0].message } };
    }
    const validated = parsed.data;

    // 2. Auth
    const supabase = await createServerClient();
    const { data: { user } } = await supabase.auth.getUser();
    if (!user) {
      return { success: false, error: { code: "UNAUTHORIZED", message: "Authentication required" } };
    }

    // 3. Rate limit
    const { success: allowed } = await rateLimit.limit(user.id);
    if (!allowed) {
      return { success: false, error: { code: "RATE_LIMITED", message: "Too many requests" } };
    }

    // 4. Appeler le service (qui verifie la policy + appelle le domain)
    const result = await listingService.create(supabase, user.id, validated);

    // 5. Log
    logger.info("listing.created", {
      listing_id: result.id,
      agency_id: validated.agency_id,
      actor: user.id,
    });

    // 6. Invalidation cache
    revalidateTag(`agency-${validated.agency_id}-listings`);

    return { success: true, data: { listing_id: result.id, status: "draft" } };
  } catch (error) {
    logger.error("listing.create.failed", { error });
    return { success: false, error: { code: "INTERNAL_ERROR", message: "An unexpected error occurred" } };
  }
}
```

### DTO Boundary Pattern (inspire de cal.com)

> **Jamais exposer les types DB (Supabase/database) directement au client.**
> Toujours transformer vers un DTO explicite a la frontiere.

```typescript
// packages/domain/listings/listing.types.ts — DTO
export interface ListingDto {
  id: string;
  title: string;
  slug: string;
  price: number;
  currency: string;
  propertyType: string;
  wilaya: string;
  coverUrl: string | null;
  agencyName: string;
  publishedAt: string | null;
}

// src/features/listings/services/listing.service.ts — mapping
function toListingDto(row: Database['public']['Tables']['listings']['Row'], translation, agency): ListingDto {
  return {
    id: row.id,
    title: translation.title,
    slug: translation.slug,
    price: Number(row.current_price),
    currency: row.currency,
    propertyType: row.property_type,
    wilaya: row.wilaya_code,
    coverUrl: null, // resolu separement
    agencyName: agency.name,
    publishedAt: row.published_at,
  };
}
```

**Regles DTO** :
- Les types DB generes (`packages/database/types/`) ne sortent JAMAIS des services
- Les composants React ne connaissent QUE les DTO
- Les server actions retournent `ActionResult<SomeDto>`, jamais `ActionResult<Database['...']>`
- Nommage : `*Dto` pour les types sortants, `*Input` pour les types entrants (Zod)

### ErrorWithCode Pattern (inspire de cal.com)

```typescript
// packages/domain/errors.ts
export class ErrorWithCode extends Error {
  constructor(
    public readonly code: string,
    message: string,
  ) {
    super(message);
    this.name = "ErrorWithCode";
  }

  static Unauthorized(message = "Unauthorized") {
    return new ErrorWithCode("UNAUTHORIZED", message);
  }
  static NotFound(message = "Not found") {
    return new ErrorWithCode("NOT_FOUND", message);
  }
  static Forbidden(message = "Forbidden") {
    return new ErrorWithCode("FORBIDDEN", message);
  }
  static Conflict(message = "Conflict") {
    return new ErrorWithCode("CONFLICT", message);
  }
  static QuotaExceeded(message = "Quota exceeded") {
    return new ErrorWithCode("QUOTA_EXCEEDED", message);
  }
}

// Usage dans les services (packages/domain)
throw ErrorWithCode.Forbidden("You are not a member of this agency");

// Conversion dans les server actions (src/features/*/actions)
try {
  const result = await listingService.create(...);
  return { success: true, data: toListingDto(result) };
} catch (error) {
  if (error instanceof ErrorWithCode) {
    return { success: false, error: { code: error.code, message: error.message } };
  }
  logger.error("unexpected_error", { error });
  return { success: false, error: { code: "INTERNAL_ERROR", message: "An unexpected error occurred" } };
}
```

### Hierarchie d'import anti-circulaire (inspire de cal.com)

```
packages/config          <- fondation, importe RIEN d'interne
  |
packages/database        <- types generes, importe config
  |
packages/domain          <- logique pure, importe database/types + config
  |
packages/security        <- guards, importe domain
  |
src/features/*/services  <- orchestration, importe domain + security + supabase
  |
src/features/*/actions   <- point d'entree, importe services + schemas
  |
src/app/                 <- pages, importe actions + components + hooks
```

**Regles** :
- **JAMAIS** d'import vers le haut (domain n'importe PAS features, features n'importent PAS app)
- **JAMAIS** d'import croise entre features (features/listings n'importe PAS features/billing)
- Si deux features doivent communiquer : extraire la logique commune dans `packages/domain`
- Enforcer via ESLint `no-restricted-imports` ou convention de review

### RBAC Permission Map (inspire de saas-starter-kit)

```typescript
// packages/security/rbac.ts
import type { AgencyRole } from "@packages/database/types";

type Permission = "create" | "read" | "update" | "delete";
type Resource = "listing" | "team_member" | "invitation" | "billing" | "settings" | "analytics";

const PERMISSIONS: Record<Resource, Record<Permission, AgencyRole[]>> = {
  listing: {
    create: ["owner", "admin", "agent", "editor"],
    read:   ["owner", "admin", "agent", "editor", "viewer"],
    update: ["owner", "admin", "agent", "editor"],
    delete: ["owner", "admin"],
  },
  team_member: {
    create: ["owner", "admin"],
    read:   ["owner", "admin", "agent", "editor", "viewer"],
    update: ["owner", "admin"],
    delete: ["owner", "admin"],
  },
  invitation: {
    create: ["owner", "admin"],
    read:   ["owner", "admin"],
    update: ["owner", "admin"],
    delete: ["owner", "admin"],
  },
  billing: {
    create: ["owner", "admin"],
    read:   ["owner", "admin"],
    update: ["owner", "admin"],
    delete: ["owner"],
  },
  settings: {
    create: ["owner", "admin"],
    read:   ["owner", "admin", "agent", "editor", "viewer"],
    update: ["owner", "admin"],
    delete: ["owner"],
  },
  analytics: {
    create: ["owner", "admin"],
    read:   ["owner", "admin", "agent"],
    update: ["owner", "admin"],
    delete: ["owner"],
  },
};

export function assertPermission(role: AgencyRole, resource: Resource, action: Permission): void {
  const allowed = PERMISSIONS[resource]?.[action] ?? [];
  if (!allowed.includes(role)) {
    throw ErrorWithCode.Forbidden(`Role '${role}' cannot '${action}' on '${resource}'`);
  }
}
```

> **Usage dans les server actions** : `assertPermission(membership.role, "listing", "create");`

### Error Boundary

```typescript
// components/ErrorBoundary.tsx
"use client";

import { Component, type ReactNode } from "react";
import { logger } from "@/lib/logger";

interface Props { children: ReactNode; fallback?: ReactNode; }
interface State { hasError: boolean; }

export class ErrorBoundary extends Component<Props, State> {
  state = { hasError: false };

  static getDerivedStateFromError() { return { hasError: true }; }

  componentDidCatch(error: Error) {
    logger.error("ui.error_boundary", { message: error.message, stack: error.stack });
    // Sentry.captureException(error);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback || <div>Une erreur est survenue. Veuillez rafraichir la page.</div>;
    }
    return this.props.children;
  }
}
```

### 00075_rpc_transactional.sql (NOUVEAU)

```sql
-- CORRIGE : SCD2 transactionnel pour eviter les race conditions

-- Changement de prix transactionnel
create or replace function change_listing_price(
  _listing_id uuid,
  _new_price numeric(14,2),
  _changed_by uuid,
  _reason text default null,
  _expected_version integer default null
)
returns uuid
language plpgsql
security definer
as $$
declare
  _now timestamptz := now();
  _new_version_id uuid;
  _current_version integer;
begin
  -- Verrouillage optimiste
  if _expected_version is not null then
    select version into _current_version
    from listings where id = _listing_id for update;

    if _current_version != _expected_version then
      raise exception 'OPTIMISTIC_LOCK_CONFLICT: listing has been modified (expected version %, got %)',
        _expected_version, _current_version;
    end if;
  else
    -- Verrouiller la ligne quand meme
    perform 1 from listings where id = _listing_id for update;
  end if;

  -- Fermer la version courante
  update listing_price_versions
  set valid_to = _now, is_current = false
  where listing_id = _listing_id and is_current = true;

  -- Ouvrir la nouvelle version
  insert into listing_price_versions (listing_id, price, valid_from, is_current, changed_by_user_id, changed_reason)
  values (_listing_id, _new_price, _now, true, _changed_by, _reason)
  returning id into _new_version_id;

  -- Mettre a jour le prix courant + incrementer version
  update listings
  set current_price = _new_price, version = version + 1
  where id = _listing_id;

  return _new_version_id;
end;
$$;

    end if;
  else
    perform 1 from listings where id = _listing_id for update;
  end if;

  -- Fermer la version courante
  update listing_status_versions
  set valid_to = _now, is_current = false
  where listing_id = _listing_id and is_current = true;

  -- Ouvrir la nouvelle version
  insert into listing_status_versions (listing_id, status, valid_from, is_current, changed_by_user_id, changed_reason)
  values (_listing_id, _new_status, _now, true, _changed_by, _reason)
  returning id into _new_version_id;

  -- Mettre a jour le statut courant + incrementer version
  update listings
  set current_status = _new_status,
      published_at = case when _new_status = 'published' then _now else published_at end,
      version = version + 1
  where id = _listing_id;

  return _new_version_id;
end;
$$;
```

### Signed Upload URL (avec validation)

```typescript
// features/media/actions/get-signed-upload-url.action.ts
const ALLOWED_TYPES = ["image/jpeg", "image/png", "image/webp"];
const MAX_SIZE_BYTES = 10 * 1024 * 1024; // 10MB

export async function getSignedUploadUrlAction(input: {
  listing_id: string;
  file_name: string;
  content_type: string;
  file_size_bytes: number;
}): Promise<ActionResult<{ signed_url: string; storage_path: string }>> {
  // Valider type MIME
  if (!ALLOWED_TYPES.includes(input.content_type)) {
    return { success: false, error: { code: "INVALID_TYPE", message: "Only JPEG, PNG and WebP images are allowed" } };
  }

  // Valider taille
  if (input.file_size_bytes > MAX_SIZE_BYTES) {
    return { success: false, error: { code: "FILE_TOO_LARGE", message: "File must be under 10MB" } };
  }

  const supabase = await createServerClient();
  const storagePath = `${agency_id}/${input.listing_id}/${input.file_name}`;

  const { data, error } = await supabase.storage
    .from("listing-media")
    .createSignedUploadUrl(storagePath);

  if (error) {
    return { success: false, error: { code: "STORAGE_ERROR", message: "Failed to generate upload URL" } };
  }

  return { success: true, data: { signed_url: data.signedUrl, storage_path: storagePath } };
}
```

### PostGIS Search + Full-Text

```sql
-- Recherche combinee : full-text + geospatial + filtres
select l.*, lt.title, lt.slug,
  ts_rank(lt.search_vector, websearch_to_tsquery('simple', $1)) as relevance
from listings l
join listing_translations lt on lt.listing_id = l.id and lt.locale = $2
where l.current_status = 'published'
  and l.deleted_at is null
  and ($1 is null or lt.search_vector @@ websearch_to_tsquery('simple', $1))
  and ($3 is null or ST_Within(l.location::geometry, ST_MakeEnvelope($3, $4, $5, $6, 4326)))
  and ($7 is null or l.listing_type = $7)
  and ($8 is null or l.property_type = $8)
  and ($9 is null or l.wilaya_code = $9)
  and ($10 is null or l.current_price >= $10)
  and ($11 is null or l.current_price <= $11)
order by
  case when $1 is not null then ts_rank(lt.search_vector, websearch_to_tsquery('simple', $1)) end desc nulls last,
  l.published_at desc
limit $12 offset $13;
```

### Stripe Webhook (Edge Function)

```typescript
// supabase/functions/stripe-webhook/index.ts
import Stripe from "stripe";
const stripe = new Stripe(Deno.env.get("STRIPE_SECRET_KEY")!);

Deno.serve(async (req) => {
  const body = await req.text();
  const sig = req.headers.get("stripe-signature")!;

  let event: Stripe.Event;
  try {
    event = stripe.webhooks.constructEvent(body, sig, Deno.env.get("STRIPE_WEBHOOK_SECRET")!);
  } catch (err) {
    return new Response("Invalid signature", { status: 400 });
  }

  // Idempotence : verifier si event.id deja traite
  const { data: existing } = await supabaseAdmin
    .from("domain_events")
    .select("id")
    .eq("aggregate_id", event.id)
    .eq("event_name", "stripe_webhook_processed")
    .single();

  if (existing) {
    return new Response(JSON.stringify({ received: true, duplicate: true }), { status: 200 });
  }

  switch (event.type) {
    case "checkout.session.completed":
      // Creer/activer subscription
      break;
    case "customer.subscription.updated":
      // Sync status
      break;
    case "customer.subscription.deleted":
      // Desactiver
      break;
  }

  // Enregistrer l'evenement comme traite
  await supabaseAdmin.from("domain_events").insert({
    aggregate_type: "stripe",
    aggregate_id: event.id,
    event_name: "stripe_webhook_processed",
    payload: { type: event.type },
  });

  return new Response(JSON.stringify({ received: true }), { status: 200 });
});
```

### Stripe Customer Idempotent (v4 — inspire de nextjs-subscription-payments)

```typescript
// features/billing/services/stripe-customer.service.ts
import Stripe from "stripe";

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);

export async function createOrRetrieveStripeCustomer(
  supabase: SupabaseClient,
  agencyId: string,
  email: string
): Promise<string> {
  // 1. Chercher dans Supabase (rapide)
  const { data: existing } = await supabase
    .from("stripe_customers")
    .select("stripe_customer_id")
    .eq("agency_id", agencyId)
    .maybeSingle();

  if (existing?.stripe_customer_id) {
    return existing.stripe_customer_id;
  }

  // 2. Chercher par email dans Stripe (reconciliation)
  const stripeCustomers = await stripe.customers.list({ email, limit: 1 });
  let customerId = stripeCustomers.data[0]?.id;

  // 3. Creer dans Stripe si inexistant
  if (!customerId) {
    const customer = await stripe.customers.create({
      email,
      metadata: { agencyId },  // Reverse lookup
    });
    customerId = customer.id;
  }

  // 4. Sync Supabase
  await supabase.from("stripe_customers").upsert({
    agency_id: agencyId,
    stripe_customer_id: customerId,
  });

  return customerId;
}
```

### Stripe Subscription Sync (v4 — inspire de nextjs-subscription-payments)

```typescript
// features/billing/services/stripe-sync.service.ts
export async function syncSubscriptionFromStripe(
  supabaseAdmin: SupabaseClient,
  stripeSubscription: Stripe.Subscription
): Promise<void> {
  const customerId = stripeSubscription.customer as string;

  // Trouver l'agence via stripe_customers
  const { data: customer } = await supabaseAdmin
    .from("stripe_customers")
    .select("agency_id")
    .eq("stripe_customer_id", customerId)
    .single();

  if (!customer) return; // Customer inconnu, ignorer

  // Trouver le plan via stripe_price_id
  const priceId = stripeSubscription.items.data[0]?.price.id;
  const { data: plan } = await supabaseAdmin
    .from("plans")
    .select("id")
    .eq("stripe_price_id", priceId)
    .single();

  // Upsert subscription (gere create ET update)
  await supabaseAdmin.from("subscriptions").upsert({
    id: stripeSubscription.id,  // PK = stripe subscription id
    agency_id: customer.agency_id,
    plan_id: plan?.id,
    stripe_subscription_id: stripeSubscription.id,
    stripe_customer_id: customerId,
    status: stripeSubscription.status as any,
    current_period_start: new Date(stripeSubscription.current_period_start * 1000).toISOString(),
    current_period_end: new Date(stripeSubscription.current_period_end * 1000).toISOString(),
    started_at: new Date(stripeSubscription.start_date * 1000).toISOString(),
    ended_at: stripeSubscription.ended_at
      ? new Date(stripeSubscription.ended_at * 1000).toISOString()
      : null,
  }, { onConflict: "stripe_subscription_id" });

  // Sync entitlements depuis le plan
  if (plan) {
    await syncEntitlementsFromPlan(supabaseAdmin, customer.agency_id, plan.id);
  }
}
```

## 12. REGLES DE SECURITE

1. **RLS deny by default** — toute table sans policy = acces interdit
2. **Server Actions uniquement** pour les mutations — jamais de mutation cote client
3. **Supabase Auth SSR** — cookies securises, pas de localStorage
4. **Verification signature Stripe** — webhook rejete si signature invalide + idempotence
5. **Pas de logique de role uniquement cote frontend** — toujours verifier cote serveur
6. **Upload via Signed URLs uniquement** — bucket prive, **validation type MIME + taille cote serveur**
7. **Donnees sensibles** (documents juridiques) : bucket prive, acces RLS, pas d'URL publique permanente
8. **PKCE pour le mobile** — stockage securise des tokens (`expo-secure-store`)
9. **Rate limiting concret** :
   - Auth : 5 tentatives / minute (Upstash)
   - Search : 30 req / minute par IP
   - Leads : 5 / heure par utilisateur
   - Actions generales : 60 / minute par utilisateur
10. **Protection CSRF** : activee nativement par Server Actions (comparaison origin/host)

## 14. TESTING & TDD

### Loi fondamentale

```
PAS DE CODE DE PRODUCTION SANS TEST ECHOUANT D'ABORD.
Code ecrit avant le test ? Le supprimer. Recommencer.
```

### Cycle Red-Green-Refactor (obligatoire)

```
RED   -> Ecrire UN test minimal montrant le comportement attendu
       -> Lancer le test
       -> VERIFIER qu'il echoue pour la bonne raison (feature manquante, pas typo)

GREEN -> Ecrire le code MINIMAL pour faire passer le test
       -> Pas de features en plus, pas de refactoring
       -> VERIFIER que le test passe ET que les autres tests passent toujours

REFACTOR -> Seulement apres GREEN
          -> Supprimer duplication, ameliorer noms, extraire helpers
          -> Les tests doivent rester verts

NEXT  -> Boucler sur le prochain test
```

### Regles de qualite des tests

| Critere | Bon | Mauvais |
|---------|-----|---------|
| Minimal | Un seul comportement par test | `test('validates email AND domain AND whitespace')` |
| Nom clair | Decrit le comportement | `test('test1')` |
| Code reel | Utilise l'implementation reelle | Mock de tout |
| Intent | Montre l'API desiree | Obscurcit le but |

### Anti-patterns de mocking (lois de fer)

1. **JAMAIS** tester le comportement du mock (asserter sur des IDs `-mock`)
2. **JAMAIS** ajouter des methodes test-only en production (cleanup, destroy...)
3. **JAMAIS** mocker sans comprendre les dependances
4. **Si le setup du mock > 50% du test** : utiliser un test d'integration a la place

### Structure des tests

```
__tests__/
  unit/           # Vitest — logique metier pure (packages/domain)
  integration/    # Vitest — server actions avec Supabase local
  e2e/            # Playwright — parcours utilisateur
  sql/            # pgTAP — RLS policies, triggers, functions, RPC
```

### Tests E2E (Playwright)

**Pattern Reconnaissance-Then-Action** (pour les apps dynamiques) :
1. Naviguer vers la page
2. Attendre `networkidle` (OBLIGATOIRE avant d'inspecter le DOM)
3. Screenshot / inspecter les selecteurs
4. Executer les actions

**Selecteurs** (ordre de preference) :
1. `getByRole()` — semantique, accessible
2. `getByText()` — lisible
3. `getByTestId()` — stable
4. CSS selector — dernier recours

**Regles Playwright** :
- Toujours headless Chromium
- Toujours fermer le browser (`browser.close()`)
- Jamais inspecter le DOM avant `networkidle`
- Utiliser `with_server.py` pour gerer le cycle de vie du serveur

### Tests critiques par module

| Module | Tests obligatoires |
|--------|-------------------|
| M01 Auth | signup, login, logout, session recovery, role check, **rate limit auth** |
| M02 Agencies | create agency, invite, accept, role change, RLS isolation, **soft-delete** |
| M03 Listings | create draft, submit for review, publish (success + refus si incomplet), price change SCD2 **transactionnel**, **verrouillage optimiste (conflict)** |
| M04 Media | upload flow, **validation type MIME refusee**, cover switch, reorder, delete + history |
| M05 Search | filtres cumules, PostGIS bounds, **full-text search**, pagination, slug par locale |
| M06 Leads | lead -> conversation, send message, **Realtime**, access denied non-participant, **read receipts** |
| M08 Billing | webhook idempotent, upgrade plan, quota enforcement, plan expired, **devise EUR** |
| M10 Moderation | **pending_review -> published**, **pending_review -> rejected**, report flow |
| RLS | Chaque table : acces autorise ET acces interdit, **profiles_public vs profiles**, **soft-delete filtre** |

### Verification avant completion (obligatoire)

**Fonction de gate** — a executer AVANT toute claim de completion :
1. **IDENTIFIER** : quelle commande prouve cette claim ?
2. **EXECUTER** : lancer la commande complete (fresh, pas de cache)
3. **LIRE** : output complet, exit code, compter les failures
4. **VERIFIER** : l'output confirme-t-il la claim ?
5. **SEULEMENT ALORS** : faire la claim avec l'evidence

**Red flags** : utiliser "devrait", "probablement", "semble" — exprimer satisfaction avant verification — committer sans verification.

### Couverture de code (v4 — inspire de cal.com)

- **80%+ de couverture pour tout nouveau code** (enforced en CI)
- Tests unitaires : viser ~100% sur `packages/domain/`
- Tests d'integration : couvrir les server actions critiques
- Tests E2E : couvrir les parcours utilisateur principaux
- Tests SQL (pgTAP) : couvrir toutes les RLS policies

```bash
# Verifier la couverture
pnpm test -- --coverage
# Seuil minimum dans vitest.config.ts :
# coverage: { branches: 80, functions: 80, lines: 80, statements: 80 }
```

### Migration Testing Strategy

```bash

supabase db reset                 # Reset + rejoue toutes les migrations
supabase test db                  # Execute les tests pgTAP

# Staging
```

## 16. WORKFLOW DE DEVELOPPEMENT

> **Cette section definit COMMENT on developpe, pas QUOI on developpe.**
> Utiliser les skills installes dans `.claude/skills/` et `.claude/agents/`.

### Cycle de developpement par feature

```
1. PLANNING     -> Utiliser le skill writing-plans
                   Creer un plan detaille dans docs/superpowers/plans/
                   Chaque tache = 2-5 minutes, une seule action

2. EXECUTION    -> Utiliser le skill subagent-driven-development
                   Un subagent frais par tache (pas d'heritage de contexte)
                   Chaque subagent suit le cycle TDD (Red-Green-Refactor)

3. REVIEW       -> Apres chaque tache :
                   a) Review spec-compliance (le code fait-il ce qui etait demande ?)
                   b) Review code-quality (le code est-il bon ?)
                   Les deux reviews DOIVENT passer avant la tache suivante

4. VERIFICATION -> Utiliser le skill verification-before-completion
                   Fonction de gate : IDENTIFIER -> EXECUTER -> LIRE -> VERIFIER
                   Pas de claim sans evidence

5. FINALIZATION -> Utiliser le skill finishing-a-development-branch
                   Verifier tests -> Presenter 4 options -> Executer le choix
```

### Planning (skill: writing-plans)

**Regles du plan** :
- Un plan = un sous-systeme independant testable
- Chaque tache est atomique (2-5 min)
- Inclure les chemins de fichiers exacts
- Inclure le code complet (pas de "ajouter la validation")
- Inclure les commandes exactes avec l'output attendu
- Appliquer DRY, YAGNI, TDD, commits frequents

**Structure du plan** :
```markdown
# Plan: [Nom de la feature]

## Objectif
[1 phrase]

## Architecture
[2-3 phrases]

## Tache 1: [Composant]
### Fichiers
- Creer: `src/features/X/actions/Y.action.ts`
- Modifier: `src/features/X/schemas/Y.schema.ts`
- Test: `__tests__/unit/X/Y.test.ts`

### Etapes
1. Ecrire le test echouant (code complet)
2. Lancer le test, verifier qu'il echoue
3. Implementer le code minimal
4. Lancer le test, verifier qu'il passe
5. Commit

## Tache 2: ...
```

### Execution avec subagents (skill: subagent-driven-development — upgraded)

> **Version upgraded depuis la branche improve-skills-from-feedback.**
> Review 2 phases separees, isolation de contexte, fixes de bugs reels.

**Par tache** :

```
1. PREPARER LE CONTEXTE
   - Extraire le texte complet de la tache (ne PAS faire lire le fichier plan)
   - Inclure : fichiers a toucher, code attendu, commandes de verification

2. DISPATCHER L'IMPLEMENTEUR
   - Subagent frais (pas d'heritage de contexte)
   - L'implementeur suit TDD : test echouant -> code minimal -> test passe -> commit
   - L'implementeur s'auto-review avant de rendre la main
   - Status possibles :
     DONE              -> passer a la review
     DONE_WITH_CONCERNS -> evaluer les concerns avant review
     NEEDS_CONTEXT      -> fournir le contexte manquant
     BLOCKED            -> escalader ou ajuster la tache

3. REVIEW SPEC-COMPLIANCE (subagent reviewer #1)
   - Verification independante : l'implementation correspond-elle a la spec ?
   - NE PAS faire confiance au rapport de l'implementeur
   - Inspecter le code ligne par ligne
   - Si problemes : l'implementeur corrige -> re-review (boucle)

4. REVIEW CODE-QUALITY (subagent reviewer #2)
   - SEULEMENT apres que spec-compliance passe
   - Verifier : qualite, patterns, tests, edge cases, securite
   - Si problemes : l'implementeur corrige -> re-review (boucle)

5. MARQUER COMPLETE
   - Les DEUX reviews doivent passer
   - Mettre a jour le TodoWrite
```

**Isolation de contexte** :
- Chaque subagent recoit un contexte cible, JAMAIS l'historique de session
- Le reviewer recoit le diff git + la spec, pas la conversation

**Regles** :
- Jamais dispatcher plusieurs subagents d'implementation en parallele
- Utiliser le modele le moins puissant par role (haiku pour mecanique, sonnet pour integration, opus pour architecture/review)
- Ne jamais faire lire le fichier plan au subagent — fournir le texte complet
- Spec-compliance AVANT code-quality (l'ordre inverse casse le process)
- Ne pas sauter les reviews meme si "c'est simple"

### Planning avec review loops (v4)

**Plan-document-reviewer** : apres chaque chunk de plan (≤1000 lignes) :
1. Dispatcher un subagent reviewer avec le chunk + le chemin de la spec
2. Le reviewer verifie : completude, alignement spec, decomposition, structure fichiers, tailles
3. Si problemes : corriger le chunk, re-dispatcher (max 5 iterations, escalader si depasse)

**Spec-document-reviewer** : apres l'ecriture d'une spec/design :
1. Dispatcher un subagent reviewer avec la spec
2. Le reviewer verifie : completude, couverture, consistance, clarte, YAGNI, frontieres architecture
3. Si problemes : corriger, re-dispatcher

### Session handoff (v4 — inspire de planning-with-files)

Pour les taches longues depassant la fenetre de contexte :
1. Creer un fichier RESUME.md avec l'etat actuel
2. Utiliser `create-handoff.sh` pour generer un prompt de reprise
3. Git checkpoint par phase (commit les progres avant le handoff)
4. La session suivante lit RESUME.md et reprend

### Agents paralleles (skill: dispatching-parallel-agents)

**Quand utiliser** :
- 3+ problemes independants sans etat partage
- Chaque probleme compris sans contexte des autres
- Les agents n'interferent pas entre eux

**Quand NE PAS utiliser** :
- Les problemes sont lies (corriger l'un pourrait corriger les autres)
- Les agents modifieraient les memes fichiers

### Debugging (skill: systematic-debugging)

**Loi fondamentale** :
```
PAS DE FIX SANS INVESTIGATION DE LA CAUSE RACINE.
Corriger un symptome = echec.
```

**4 phases (completer chacune avant la suivante)** :

1. **Investigation** : lire les erreurs, reproduire, verifier les changements recents, tracer le flux de donnees
2. **Analyse de pattern** : trouver des exemples fonctionnels similaires, comparer, lister chaque difference
3. **Hypothese** : formuler UNE hypothese, tester minimalement, verifier
4. **Implementation** : creer un test echouant, implementer UN fix, verifier

**Regle des 3 echecs** : si 3+ fixes echouent → arreter, questionner l'architecture, discuter avec l'humain.

### Code Review (skill: requesting-code-review)

**Obligatoire** :
- Apres chaque tache en subagent-driven-development
- Apres chaque feature majeure
- Avant merge vers main

**Process** :
1. Obtenir les SHAs git (base et head)
2. Dispatcher le subagent code-reviewer avec le contexte
3. Agir sur le feedback : Critical = fix immediat, Important = fix avant de continuer, Minor = noter

### Fin de branche (skill: finishing-a-development-branch)

**Process** :
1. Lancer les tests → doivent passer
2. Presenter exactement 4 options :
   - Merge local vers la branche de base
   - Push + creer une PR
   - Garder la branche telle quelle
   - Abandonner le travail
3. Executer le choix
4. Nettoyer le worktree si necessaire

## 17. DESIGN SYSTEM AVANCE (UI/UX)

> **Complement a la section 8. Regles extraites du skill ui-ux-pro-max.**
> **Utiliser le skill `ui-ux-pro-max` pour toute creation de composant UI.**

### Accessibilite (CRITIQUE — non-negociable)

- Contraste couleur : minimum **4.5:1** texte normal, **3:1** grand texte
- Focus visible : ring 2-4px sur tous les elements interactifs
- Alt text : descriptif pour toutes les images significatives
- Aria labels : `aria-label` sur les boutons icon-only
- Navigation clavier : tab order = ordre visuel, support clavier complet
- Labels de formulaire : `<label htmlFor="">` (jamais placeholder-only)
- Hierarchie headings : h1 -> h6 sequentiel, pas de saut
- Couleur pas seule : ne pas transmettre d'info par couleur uniquement (ajouter icone/texte)
- Reduced motion : respecter `prefers-reduced-motion`
- Skip links : "Aller au contenu principal" pour clavier

### Touch & Interaction (CRITIQUE)

- Cible tactile : minimum **44x44pt**, etendre la zone de hit si necessaire
- Espacement tactile : minimum 8px entre les cibles
- Hover vs tap : click/tap pour les actions principales, jamais hover-only
- Boutons loading : desactiver pendant async, montrer spinner
- Feedback erreur : message clair pres du champ concerne
- Cursor pointer : sur tous les elements cliquables

### Animation (MOYEN)

- **Duree** : 150-300ms micro-interactions, transitions complexes <= 400ms, jamais > 500ms
- **Transform only** : utiliser transform/opacity uniquement, eviter width/height/top/left
- **Loading** : skeleton ou indicateur de progression si chargement > 300ms
- **Signification** : chaque animation exprime une relation cause-effet
- **Sortie rapide** : animations de sortie = 60-70% de la duree d'entree
- **Stagger** : echelonner l'entree des items liste/grille par 30-50ms
- **Interruptible** : les animations doivent etre interruptibles, jamais bloquer l'input
- **Page load** : staggered reveals pour l'impact (entree echelonnee des elements)

### Formulaires (MOYEN)

- Labels visibles par input (jamais placeholder-only)
- Erreur en dessous du champ concerne
- Submit : loading puis succes/erreur
- Champs requis : marquer avec asterisque
- Etats vides : message utile + action
- Toasts : auto-dismiss 3-5s
- Confirmation : avant actions destructrices
- Helper text : texte persistant sous les inputs complexes
- Disabled : opacite reduite (0.38-0.5) + cursor change
- Disclosure progressive : reveler les options complexes progressivement
- Validation inline : sur blur (pas keystroke), erreur apres que l'utilisateur a fini
- Input types : email, tel, number pour le bon clavier mobile
- Password toggle : show/hide pour les mots de passe
- Messages d'erreur : cause + comment corriger
- Focus management : apres erreur submit, auto-focus le premier champ invalide

### Navigation

- Bottom nav : max 5 items, labels + icones
- Drawer/sidebar : pour navigation secondaire, pas actions principales
- Back : previsible et consistant, preserver scroll/state
- Deep linking : tous les ecrans cles accessibles via URL
- Labels de nav : icone + texte toujours
- Etat actif : location actuelle visuellement mise en evidence
- Hierarchie : navigation primaire (tabs/bottom) vs secondaire (drawer) clairement separee

### Performance UI

- Images : WebP/AVIF avec lazy loading
- Declarer width/height sur les images (eviter layout shift)
- Skeleton screens pour loading > 300ms
- Virtualiser les listes 50+ items
- Un seul scroll primaire par viewport (eviter scroll imbrique)

### Anti-patterns de design a eviter

- **Esthetique AI generique** : polices overused (Arial, Roboto, system-ui seul), gradients violets sur blanc, layouts predictibles
- **Noms de classes dynamiques** : Tailwind ne peut pas les detecter — eviter
- **Couleurs hardcodees** : toujours `var(--token)`, jamais hex brut
- **Largeurs arbitraires** : utiliser les tokens de design
- **Hover-only info** : inaccessible sur mobile

### Gestion des quotas

```typescript
// Verifier le quota avant l'appel
async function checkAiQuota(supabase, agencyId: string): Promise<boolean> {
  const { data: subscription } = await supabase
    .from("subscriptions")
    .select("plan:plans(max_ai_jobs)")
    .eq("agency_id", agencyId)
    .eq("status", "active")
    .single();

  const { count } = await supabase
    .from("ai_jobs")
    .select("*", { count: "exact", head: true })
    .eq("agency_id", agencyId)
    .gte("created_at", startOfMonth());

  return count < subscription.plan.max_ai_jobs;
}
```
